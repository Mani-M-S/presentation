Sécurité

OWASP Top 10 for LLM — Implémenté, pas juste listé

Chaque vulnérabilité est traitée par une mesure concrète dans le code

LLM01 — INJECTION DE PROMPT
Comment : check_query_appropriateness() — LLM classificateur dédié répond SAFE ou UNSAFE avant tout traitement. Double passe (brut + normalisé).
LLM02 — SORTIES NON SÉCURISÉES
Comment : bleach.clean() sanitise toutes les sorties LLM (anti-XSS). Limites max_tokens par rôle : 800 / 60 / 200.
LLM03 — CHAÎNE D'APPROVISIONNEMENT
Comment : Dépendances épinglées avec == dans requirements.txt — aucune mise à jour silencieuse vers une version vulnérable.
LLM05 — MOINDRE PRIVILÈGE
Comment : PII (nom, email, tél) stockés côté serveur dans Redis — jamais transmis par le client. Un attaquant intercepté ne voit pas les données personnelles.
LLM06 — DIVULGATION D'INFORMATIONS
Comment : Clés API dans .env jamais dans le code, listées dans .gitignore. Fichier env.example avec valeurs vides fourni aux contributeurs.
LLM07 — VALIDATION DES ENTRÉES
Comment : Pydantic : min_length=1, max_length=2000. Timeout LLM 10 s. Requête invalide → HTTP 422 avant d'atteindre le chatbot.
LLM09 — DÉSINFORMATION / HALLUCINATION
Comment : RAG + instruction "Reply ONLY using context". Flag no_results:true quand ChromaDB retourne vide → pas de réponse inventée.
LLM10 — RATE LIMITING & DoS
Comment : slowapi : 5 req/min sur /chat → HTTP 429. Protège contre les attaques DoS et les abus de coût API.
